# Productization Runbook

Goal: turn programs, research insights, workflows, and tools into sellable products.

## Product opportunity detection rules
Productize when item is:
- repeatedly requested (2+ clients/use-cases),
- outcome-linked (clear before/after value),
- teachable/documentable,
- low customization for core 80%.

## Product types taxonomy
- Template pack
- SOP bundle
- Automation spec
- Research brief
- Toolkit
- Service package

## Vertical adaptation rules
- Keep core mechanism constant.
- Swap language, examples, compliance constraints, and KPIs by industry.
- Maintain one shared core + vertical overlays (not separate products from scratch).

## Pricing logic
- **Low ticket**: self-serve, minimal support, high repeatability.
- **Mid ticket**: guided implementation + light customization.
- **High ticket**: strategic customization, higher risk/impact, hands-on delivery.

## Effort vs leverage scoring (1–5 each)
- Build effort
- Delivery effort
- Reusability
- Margin potential
- Outcome strength
Prioritize high leverage + high reusability + strong outcome.

## Repeatability detection
Reusable if:
- same workflow solves similar problems across 2+ contexts,
- inputs/outputs can be standardized,
- exceptions are manageable via add-on SOPs.

## Differentiation layer
Define what is uniquely better:
- speed to outcome,
- implementation clarity,
- risk controls,
- vertical-specific playbooks,
- measurable KPI guarantees/targets.

## Deliverable packaging structure
- Product one-pager (problem, user, outcome)
- Scope + exclusions
- Core assets (templates/SOP/specs)
- Implementation path
- KPI/ROI measurement plan
- Support model + upgrade path

## Example product output structure
- Product name + ICP
- Problem solved
- Included assets
- Delivery format
- Timeline
- Pricing tier
- Success metrics
- Case/example scenario

## Product maturity levels
- **Prototype**: tested once, unstable docs.
- **v1**: repeatable with guidance.
- **Standardized**: consistent outcomes, clear SOPs.
- **Scalable**: delegated delivery, low founder dependence.

## Do-not-productize rules
Do not productize when:
- one-off custom edge case,
- weak/unclear outcomes,
- heavy dependence on founder intuition only,
- legal/compliance risk too high for repeat delivery,
- evidence of repeatability is below threshold.

## Operating bias
- Prefer simple products first.
- Bias toward reusable assets.
- Bias toward systems over one-off work.
